The problem I saw

Every Parent-Teacher Conference, our school turned into a data collection emergency.

As a form teacher, I would spend days before each conference opening School Cockpit for CCA records, RPMS for existing reports, and five separate Excel files from subject teachers — just to piece together a complete picture of one student. Nothing talked to anything else.

Subject teachers found the comment-writing process tedious and repetitive. Leadership had no single view of which students needed urgent attention. The data existed — it was just trapped in silos.

01
Fragmented data sources
Grades in RPMS, CCA in Cockpit, teacher comments in individual Excel files. No single source of truth.
02
Manual, repetitive work
Teachers copy-pasting scores into comment templates. Form teachers manually consolidating 20+ files before each conference.
03
No early warning system
At-risk students only identified reactively — after exam results, not before. Leadership lacked predictive visibility.

System design rationale

Key design decisions

Every choice in this system was deliberate. Here's the reasoning behind the four decisions that matter most.

Data integration
Student ID as the universal join key
Names get misspelled. Classes change. Student ID is the only identifier that stays stable across School Cockpit, RPMS, and every teacher's Excel. Designing around this key means any new data source can be integrated without restructuring the pipeline.
Privacy by design
Client-side Excel parsing — no server required
SheetJS parses uploaded files entirely in the browser. Student data never leaves the teacher's device. This isn't just good practice — it removes a whole category of PDPA risk before it can arise, and means the system can be deployed without school IT needing to provision a secure data server.
AI governance
Human-in-the-loop before every send
The AI generates parent letters, but a teacher must review and explicitly approve before anything is sent. The audit log records who generated each letter, when, and who approved it. This is not just a UX choice — it's a governance requirement. An AI-generated letter sent without review is an institutional liability.
Risk model design
Weighted scoring over raw grades
A student failing one subject but with 98% attendance and active CCA participation has a very different risk profile from a student with the same grade but chronic absenteeism. The risk score weights attendance (30%) and socio-behavioural signals (20%) alongside academics (50%) — informed by seven years of observing which signals actually predict long-term student outcomes.
Production roadmap
n8n automation for full deployment
In production, the manual Excel upload step would be replaced by an n8n workflow: Google Classroom triggers a webhook → Python script pulls updated scores → student_analytics updates automatically → high-risk students are flagged to the form teacher every Monday morning. The ReportReady interface stays the same; only the data ingestion becomes fully automated.
Institutional context
Built by someone who's used these systems
RPMS is not just an acronym in a job description — I've filled it in. I know why teachers find the comment process draining, what form teachers actually need to see before a conference, and what makes a parent letter land well. Domain knowledge shapes better systems than technical skill alone.
